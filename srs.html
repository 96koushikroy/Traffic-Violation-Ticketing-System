1. Introduction

1.1 Purpose

The product deals with the Traffic violation; where the traffic police fines the driver with this software and the driver can log into the software and check his fine and due date. This is the initial version of the SRS.And has scopes in government in traffic control centres.

1.2 Document Conventions

While writing this SRS, IEEE SRS formatting was used. All the six sections in this SRS has a bold font , and are larger than the subsections. Whereas, at the same time, the subsections also have bold fonts and are larger than the description of each subsections.

1.3 Intended Audience and Reading Suggestions

The readers that the document is intended for the developes , marketing staff , testers and users(police and general citizen).This SRS contains the descriptions of the users clases and interface, products, design and implementation Constraints, interfaces of the hardware , software and communications, the useclass of different systems and other non functional requirements.

1.4 Product Scope

The software deals with the implementation on a smart phone application which allows the traffic police and general citizens to deal with the process of ticketing after the violation of traffic. The police takes a picture of the lisence and vehicle registration plate by the use of the API , and fines the driver using the app; while the driver can log into the software and input the lisence number to check the due date and the fine they have to pay.The goal is to make the whole system of traffic violation ticketing easier than before , for both the police and driver. The benefits of this system is that it's more accessible, the users(driver) will not need to stand in line for hours to get their schedule to pay the fine; whereas the police will not need to keep tracks of papers and send it to the traffic office everyday to get the cases registered, it will get registered automatically.

5. Other Nonfunctional Requirements

5.1 Performance Requirements

We recommend to use a new generation Android phone to ensure smooth and lag free navigation in the mobile app. The camera of the Android phone for the Police should be of better quality to help during Image Processing. The web application should operate without any performance issues in any normal condition. When making API calls from both the Web app and the mobile app the duration of the response depends on the internet connection latency with the API Server. This factor will also affect when the API server will request the database server for data or manipulate data.

5.2 Safety Requirements

The information will be encrypted and will be transmitted securely to and from the server. The passwords will be encrypted using salted SHA1 algorithm which will ensure user account security.

5.3 Security Requirements

No specific security requirement just that the users should maintain their own security physically around them to avoid their accounts being compromised.

5.4 Software Quality Attributes

  • Easy To Use

    • The easy to use interfaces can be very helpful for the users to work and navigate through the app.

  • Credential Security

    • Ensure that the users credentials are throughly secured with secure data transfer protocols and encrypted on the server.

5.5 Business Rules

The Administrator Accounts will be created by default. The Administrator in this system is responsible for creating the Police Accounts. So, the Police Accounts mainly function after they have been created by the Administrator. On the other hand the User Accounts are independent of any other users/rules, as they can create the accounts themselves. The tickets that the users will see in their account will show up if the police adds a ticket for that particular users Driving License Number.
